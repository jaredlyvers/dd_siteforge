# Preview Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Press `p` to preview the current page in the system browser. The flow reuses the validation gate + export pipeline already shipped in §2 (validate → resolve `export_dir` → render `site_to_dir` → copy source images), then spawns the OS-specific opener (`xdg-open` on Linux, `open` on macOS, `start` on Windows) on `<export_dir>/<current-page-slug>.html`.

**Architecture:** One new helper method `begin_preview_flow()` that mirrors the structure of `begin_export_flow` but routes the success branch to a `commit_preview_to(rel: String)` instead of `commit_export_to`. The first-time path prompt has its own `Modal::PreviewPathPrompt { path: String }` so the post-prompt commit knows to open the browser (re-using `Modal::ExportPathPrompt` would lose that intent on the modal-event-handler side). Browser launch is a free fn `open_in_browser(path) -> io::Result<()>` that dispatches per `cfg!(target_os = ...)`. Failure to spawn the opener surfaces as a Warning toast but the export itself is still considered successful (matches spec §6).

**Tech Stack:** Rust 2024, `std::process::Command`, no new deps.

**Plan covers spec section §6 only.** Image-picker (§3) deferred — preview just uses whatever paths the components already point to (e.g. `/assets/images/...`); copying `./source/images/` is already wired in §2.

```rust
/// Spawn the OS-default opener on the given file path. Returns the spawn
/// error if the command can't be invoked. The browser may take time to
/// open after this returns; we don't wait.
fn open_in_browser(path: &std::path::Path) -> std::io::Result<()> {
    use std::process::Command;
    #[cfg(target_os = "linux")]
    let mut cmd = {
        let mut c = Command::new("xdg-open");
        c.arg(path);
        c
    };
    #[cfg(target_os = "macos")]
    let mut cmd = {
        let mut c = Command::new("open");
        c.arg(path);
        c
    };
    #[cfg(target_os = "windows")]
    let mut cmd = {
        let mut c = Command::new("cmd");
        c.args(["/C", "start", ""]).arg(path);
        c
    };
    #[cfg(not(any(target_os = "linux", target_os = "macos", target_os = "windows")))]
    let mut cmd = {
        return Err(std::io::Error::new(
            std::io::ErrorKind::Unsupported,
            "no known browser opener for this target",
        ));
    };
    cmd.spawn()?;
    Ok(())
}
```

```rust
fn handle_preview_path_prompt_event(&mut self, key: event::KeyEvent) -> Option<ModalResult> {
    use crossterm::event::KeyCode;
    let path = if let Some(Modal::PreviewPathPrompt { path }) = self.modal.take() {
        path
    } else {
        return Some(ModalResult::CloseCancel);
    };
    match key.code {
        KeyCode::Esc => {
            self.push_toast(ToastLevel::Info, "Preview cancelled.");
            Some(ModalResult::CloseCancel)
        }
        KeyCode::Enter => self.commit_preview_path_from_prompt(path),
        KeyCode::Char('s') if key.modifiers.contains(KeyModifiers::CONTROL) => {
            self.commit_preview_path_from_prompt(path)
        }
        KeyCode::Backspace => {
            let mut new_path = path;
            new_path.pop();
            self.modal = Some(Modal::PreviewPathPrompt { path: new_path });
            Some(ModalResult::Continue)
        }
        KeyCode::Char(c) if !key.modifiers.contains(KeyModifiers::CONTROL) => {
            let mut new_path = path;
            new_path.push(c);
            self.modal = Some(Modal::PreviewPathPrompt { path: new_path });
            Some(ModalResult::Continue)
        }
        _ => {
            self.modal = Some(Modal::PreviewPathPrompt { path });
            Some(ModalResult::Continue)
        }
    }
}

fn commit_preview_path_from_prompt(&mut self, path: String) -> Option<ModalResult> {
    let trimmed = path.trim();
    if trimmed.is_empty() {
        self.push_toast(ToastLevel::Warning, "Preview path required.");
        self.modal = Some(Modal::PreviewPathPrompt { path });
        Some(ModalResult::Continue)
    } else {
        self.commit_preview_to(trimmed.to_string());
        Some(ModalResult::CloseSuccess)
    }
}
```

- [ ] **Step 1: Replace the `commit_preview_to` stub**

```rust
/// Resolve `rel` against the site JSON's directory (or cwd if no path),
/// run the renderer, copy source images, persist `export_dir`, then spawn
/// the system browser on the current page's HTML. Browser-spawn errors
/// surface as a Warning toast but the export itself is still treated as
/// a success.
fn commit_preview_to(&mut self, rel: String) {
    use std::path::{Path, PathBuf};
    let normalized = normalize_relative_path(&rel);
    let base = self
        .path
        .as_ref()
        .and_then(|p| p.parent().map(PathBuf::from))
        .unwrap_or_else(|| PathBuf::from("."));
    let out = base.join(Path::new(&normalized));

    if let Err(e) = crate::renderer::render_site_to_dir(&self.site, &out) {
        let msg = format!("Preview failed: {}", e);
        self.push_toast(ToastLevel::Warning, msg);
        return;
    }
    self.site.export_dir = Some(normalized.clone());
    self.copy_source_images_to(&base, &out);

    let slug = self.current_page_slug_for_preview();
    let target = out.join(format!("{}.html", slug));
    let display = display_relative_path(&base, &out, &normalized);
    let count = self.site.pages.len();
    self.push_toast(
        ToastLevel::Success,
        format!("Exported {} page(s) to {}", count, display),
    );
    match open_in_browser(&target) {
        Ok(()) => {
            self.push_toast(
                ToastLevel::Info,
                format!("Opening {} in browser…", target.display()),
            );
        }
        Err(e) => {
            self.push_toast(
                ToastLevel::Warning,
                format!("Browser open failed: {}", e),
            );
        }
    }
}

/// The slug of the current selection, falling back to the first page when
/// the selection is somehow out of bounds.
fn current_page_slug_for_preview(&self) -> String {
    let idx = self.selected_page.min(self.site.pages.len().saturating_sub(1));
    self.site
        .pages
        .get(idx)
        .map(|p| p.slug.clone())
        .unwrap_or_else(|| "index".to_string())
}
```

- [ ] **Step 2: Add `begin_preview_flow`**

```rust
/// Entry point for the `p` key. Mirrors `begin_export_flow` but routes the
/// success path to `commit_preview_to` so the browser opens after rendering.
fn begin_preview_flow(&mut self) {
    let errors = crate::validate::validate_site(&self.site);
    if !errors.is_empty() {
        self.modal = Some(Modal::ValidationErrors {
            errors,
            scroll_offset: 0,
        });
        return;
    }
    match self.site.export_dir.clone() {
        Some(dir) if !dir.trim().is_empty() => {
            self.commit_preview_to(dir);
        }
        _ => {
            self.modal = Some(Modal::PreviewPathPrompt {
                path: "./web/".to_string(),
            });
        }
    }
}
```

## Self-Review Notes

- **Spec coverage (§6):**
  - `p` key on a row that maps to a specific page — Task 4 (binding is global; the slug used is the selected page's slug, so any time you have a page selected it works as expected).
  - Runs full export flow including validation gate — Task 3 `begin_preview_flow`.
  - Opens system browser on `<export_dir>/<current-page-slug>.html` — Task 3 `commit_preview_to`.
  - xdg-open / open / start — Task 1.
  - Browser spawn failure surfaces in status — implemented as Warning toast (toasts replaced status-bar success/warning earlier in this rollout).

- **Type consistency:** `open_in_browser`, `Modal::PreviewPathPrompt`, `begin_preview_flow`, `commit_preview_to`, `current_page_slug_for_preview`, `commit_preview_path_from_prompt`, `render_preview_path_prompt`, `handle_preview_path_prompt_event` referenced consistently across all 5 tasks.

- **Caveats:**
  - Browser launch is fire-and-forget (`spawn`). We don't wait or track exit. If a user wants to wire WebSocket-driven live reload, that's a separate ticket — spec §6 explicitly scopes to "minimal".
  - `xdg-open` requires a desktop environment; headless Linux servers won't have it. The fallback Warning toast tells the user what failed.
  - Tests do NOT actually launch the browser; they cover the dispatch (validate → prompt OR commit) and the slug computation. The `commit_preview_to` happy-path could attempt to spawn the opener during a hypothetical future test that provides a saved `export_dir`, which would leak browser windows in CI — I deliberately don't add such a test.
